# Contributing Guidelines

Thank you for your interest in contributing to this program.  
This repository is used for learning, collaboration, and real-world project development. Contributors are expected to follow professional software engineering practices and maintain a collaborative environment.

---

## Who Can Contribute?
- Students
- Interns
- Open Source Contributors
- Researchers
- Industry Professionals
- Community Members

---

## Before You Start
- Git installed
- GitHub account created
- Development environment configured
- Read the project documentation
- Completed onboarding activities (if applicable)

Recommended reading:
- README.md
- PROJECTS.md
- TRAINING.md
- CODE_OF_CONDUCT.md

---

## Contribution Workflow

### Step 1: Find an Issue
- Browse available issues and identify a task suitable for your skill level.

Issue labels:
- good-first-issue – Suitable for beginners
- documentation – Documentation related
- enhancement – New feature
- bug – Bug fix
- research – Research activity
- help-wanted – Community support requested

---

### Step 2: Request Assignment
- Comment on the issue:  
  `I would like to work on this issue.`
- Wait for mentor approval before starting work.

---

### Step 3: Fork Repository
- Create your own fork of the repository.
- Example:  
  `git clone https://github.com/<your-account>/<repository>.git`

---

### Step 4: Create a Branch
- Never commit directly to the `main` branch.
- Use the following naming convention:
  - feature/issue-101-add-authentication
  - bugfix/issue-205-fix-login
  - docs/update-installation-guide
- Example:  
  `git checkout -b feature/issue-101-add-authentication`

---

### Step 5: Implement Changes
- Follow project coding standards
- Keep commits small and meaningful
- Update documentation when required
- Add tests when applicable

---

### Step 6: Commit Changes
- Write clear commit messages

Good examples:
- feat: add user authentication service
- fix: resolve function deployment issue
- docs: update installation instructions
- test: add unit tests for scheduler

Avoid:
- update
- changes
- fixed issue

---

### Step 7: Push Changes
- `git push origin <branch-name>`

---

### Step 8: Create Pull Request
- Create a Pull Request (PR) against the appropriate branch.
- Example PR Title:  
  `feat: add node registration service`

---

## Pull Request Requirements
- Summary – What was implemented
- Motivation – Why the change was needed
- Testing – How the changes were tested
- Documentation – Any updated docs
- Screenshots – When applicable

---

## Code Review Process
- All contributions are reviewed before merging.
- Reviewers may request:
  - Code improvements
  - Additional tests
  - Documentation updates
  - Security considerations
  - Performance improvements
- Contributors should address review comments respectfully and promptly.

---

## Coding Standards
- Write readable code
- Follow project conventions
- Avoid duplicated logic
- Use meaningful names
- Keep functions focused

---

## Documentation Standards
- Purpose
- Usage instructions
- Configuration details
- Examples

---

## Security Guidelines
- Never commit passwords, API keys, secrets, tokens, certificates, or personal information.
- Use environment variables instead.  
  Example:  
  `API_KEY=${API_KEY}`

---

## Branch Protection Rules
- The `main` branch is protected.
- Direct commits to `main` are not permitted.
- Changes must go through:
  - Pull Request
  - Code Review
  - Approval
  - Merge

---

## Contributor Levels

Level 1 – Trainee Engineer
- Learning GitHub workflow
- Working on beginner issues
- Activities: Documentation, Testing, Small enhancements

Level 2 – Associate Contributor
- Working on project tasks
- Participating in reviews
- Activities: Feature implementation, Bug fixes, Technical documentation

Level 3 – Project Contributor
- Owning project modules
- Supporting junior contributors
- Activities: Design discussions, Feature ownership, Review participation

Level 4 – Technical Lead Intern
- Leading small teams
- Reviewing pull requests
- Coordinating contributors
- Activities: Technical leadership, Mentoring, Architecture discussions

## Reporting Issues
